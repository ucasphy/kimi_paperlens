# 学习模式 — Deep Learn

## 目标

让非专业背景的读者也能深度理解论文，并能指导实际场景落地。

## 核心原则：大白话

这个模式的灵魂是**大白话**。所有解释都遵循：

1. **先给直觉，再给细节** — 先说核心要点，再展开技术细节
2. **类比精准** — 只对真正抽象难懂的概念使用类比，每个概念最多一个比喻且一击即中；简单概念直接用准确的技术语言解释
3. **重构而非翻译** — 不是把英文论文翻译成中文，而是用自己的话重新讲一遍
4. **承认边界** — 对论文没说清楚的地方，坦诚标注"推测"或"论文未说明"
5. **避免比喻堆叠** — 不连续使用「打个比方」「简单来说」「就像」「换句话说」等过渡语；一句话能说清的不需要比喻

## 增量学习笔记机制

学习模式的核心体验：**边学边存，随时可读**。

### 笔记文件

路径：`paper-notes/<name>/deep-learn.md`

### 保存规则

1. **Step 1（速览）完成后**：创建文件，写入速览内容作为笔记的第一部分
2. **每完成一个 Step**：把该 Step 的输出追加到笔记文件中
3. **用户通过自然语言要求更新时**：随时修改笔记内容（增删改均可）
4. **每次保存后**：简短提示用户「已更新学习笔记 → paper-notes/<name>/deep-learn.md」
5. **断点恢复标记**：每次追加内容后，在文件末尾写入进度标记 `<!-- paper-lens:progress:stepN -->`（N 为当前完成的步骤编号）。恢复学习时，检测文件末尾的标记，跳到对应步骤的下一步继续

### 笔记文件结构

```markdown
# <论文标题> — 学习笔记

> 论文：<英文标题>
> 生成日期：<日期>
> 模式：paper-lens 学习模式

---

## 速览
<Step 1 的内容>

## 术语表
<Step 2-3 的内容，每个术语的解释>

## 核心方法拆解
<Step 4 的内容>

## 公式解读
<Step 5 的内容（如有）>

## 附录精华
<Step 6 的内容>

## 实践指导
<Step 7 的内容>

## 自由笔记
<用户在对话中通过自然语言补充的内容>
```

### 自然语言更新

用户在任何阶段都可以用自然语言请求修改笔记，例如：
- "把刚才那个解释加到笔记里"
- "帮我在笔记里补充一段关于 XXX 的理解"
- "笔记里那个 Pass@1 的解释改一下，加上我自己的理解：..."
- "删掉笔记里附录那部分，我不需要"

收到这类请求时，直接用 Edit 工具修改笔记文件，然后继续当前流程。

## 术语格式规范

**所有英文术语首次出现时，必须在后面加中文注释**，格式：

- 行内：`Pass@1（单次通过率）`、`Active Learning（主动学习）`
- 列表项标题：`**Pass@1（单次通过率）**`
- 括号内用中文短语，不超过 8 个字，优先用已有的通用译法

## 流程（多轮交互）

### Step 1: 速览内容

先检查 `paper-notes/<name>/speed-read.md` 是否已存在。若存在，读取其内容作为速览部分直接复用，不重复执行速览模式。若不存在，执行速览模式的全部内容。

**断点恢复**：如果 `paper-notes/<name>/deep-learn.md` 已存在，读取文件末尾的进度标记 `<!-- paper-lens:progress:stepN -->`。若检测到标记，跳过已完成的步骤，从 step N+1 继续，并提示用户：「检测到上次学习进度（已完成到 Step N），从 Step N+1 继续。」

**输出后**：
1. 创建笔记文件 `paper-notes/<name>/deep-learn.md`，写入速览内容
2. 在文件末尾追加 `<!-- paper-lens:progress:step1 -->`
3. 提示：「速览部分已完成，已保存到学习笔记。接下来我会提取论文中的专业术语，你来选择想深入了解的。」

---

### Step 2: 名词提取与交互确认

**自动扫描论文全文，提取专业术语，直接用 AskUserQuestion 让用户选择。**

**提取规则**：
- 论文专有概念：论文中首次定义的术语、自创缩写、特有的架构/模块名称
- 领域通用概念：论文引用但未定义的技术术语、工具名称、数据集名称、评估指标名称
- 排除：众所周知的概念（如 AI、深度学习、Python 等）
- **所有英文术语后必须带中文注释**

#### 交互选择（一步完成，不要拆成多步）

提取完术语后，**直接使用一次 AskUserQuestion 工具**，同时包含两个问题：

**第一个问题**（多选）：论文专有概念

```
问题：论文专有概念——你想了解哪些？（可多选，「其他」可补充任何你想问的）
选项：（把专有概念按相关性分组，每组作为一个选项，控制在 4 个以内）
  - "全部解释（推荐）" — 论文核心概念，建议都了解
  - "<概念组A>（<中文>）" — <一句话说明>
  - "<概念组B>（<中文>）" — <一句话说明>
  - "<概念组C>（<中文>）" — <一句话说明>
multiSelect: true
```

**第二个问题**（多选）：领域通用概念

```
问题：领域通用概念——哪些你想深入了解？（可多选，「其他」可输入论文中你看到的任何想了解的内容）
选项：（从提取的领域通用概念中选取关键的，按相关性分组，控制在 4 个以内）
  - "<概念/概念组A>（<中文>）" — <一句话说明>
  - "<概念/概念组B>（<中文>）" — <一句话说明>
  - "<概念/概念组C>（<中文>）" — <一句话说明>
  - "<概念/概念组D>（<中文>）" — <一句话说明>
multiSelect: true
```

**分组原则**：
- AskUserQuestion 每个问题最多 4 个选项，所以需要按相关性把术语分组
- 分组时把紧密关联的概念放在一起，如"三种任务构造策略"可以作为一组
- 每个选项的 label 要简洁，description 里给出具体包含哪些术语
- **不要先问"全部/让我选择/跳过"再出具体选项**——直接一步到位

**AskUserQuestion 自带「其他」选项**，用户可以自由输入：
- 术语（如 "Docker"、"ICL"）→ 加入待解释列表
- 问题（如 "为什么要用主动学习？"）→ 在 Step 3 末尾回答
- 模糊内容（如 "那个五步流水线"）→ 理解意图，给出对应解释
- 所有自定义内容都纳入笔记

### Step 3: 名词解释

对确认的术语，用表格形式紧凑呈现。

**输出格式**：

```markdown
### 一、论文专有概念

| 术语 | 定义 | 解释 | 论文角色 |
|------|------|------|----------|
| **<术语>（<中文>）** | <一句话定义> | <一句话解释，简单概念直接技术说明，复杂概念可用一个精准类比> | <在论文中的角色> |
| ... | ... | ... | ... |

> **<复杂术语>** 补充：<仅对 2-3 个确实需要展开的复杂术语补充 1-2 句说明>

### 二、领域通用概念

| 术语 | 定义 | 解释 | 论文角色 |
|------|------|------|----------|
| ... | ... | ... | ... |
```

**表格规则**：
- 「定义」列：一句话，≤20 字，技术准确
- 「解释」列：一句话大白话，≤30 字；简单概念不用比喻，复杂概念可用一个类比
- 「论文角色」列：一句话说明在本论文中的具体作用
- 每个表格之后，仅对 2-3 个确实复杂的术语用 blockquote 补充说明（可选）
- 论文专有概念和领域通用概念分开两个表格

**风格示例**：

```markdown
### 一、论文专有概念

| 术语 | 定义 | 解释 | 论文角色 |
|------|------|------|----------|
| **Pass@1（单次通过率）** | 模型一次生成就答对的概率 | 只给一次机会，对了就得分 | 衡量 Bug 修复/功能实现的核心指标 |
| **ISR（实例成功率）** | 全部检查项通过才算成功的实例占比 | 100 道题全答对才及格 | 主指标，所有模型仅 9-28% |

> **Pass@1** 补充：50% 意味着 100 道题大约一次做对 50 道。分数越高说明模型越可靠——不需要反复尝试。
```

**如果用户有自定义问题**，在术语表完成后，单独用 `## 你的问题` 板块回答。

**输出完成后**：
1. 将术语表和问题回答追加到笔记文件
2. 提示：「术语解释已完成并保存到笔记。接下来拆解核心方法。如果你还有想了解的术语，随时告诉我，我会补充到笔记里。」

---

### Step 4: 核心方法的大白话拆解

**把论文方法论的核心逻辑用日常语言重新讲一遍。**

```markdown
## 核心方法拆解

### 整体思路（大白话版）

<用 3-5 句话，像给朋友讲故事一样说清楚这篇论文到底做了什么。>

### 详细拆解

#### <步骤/模块 1 名称>

**做了什么**：<一句话>

**大白话解释**：
<用日常语言解释，可以用类比。>

**流程图**：
```
<用 ASCII art 画出关键流程>
```

#### <步骤/模块 2 名称>
...（同上格式）

### 关键设计选择

> **为什么选择 A 而不是 B？**
> <解释设计背后的权衡，用通俗语言>
```

**要求**：
- 必须包含一个 ASCII art 流程图，帮助可视化理解
- "关键设计选择"部分分析作者为什么这样做，而不只是描述做了什么
- 如果论文有多个核心模块，逐个拆解，但每个控制在 200 字以内

**输出完成后**：追加到笔记文件，提示已保存。

### Step 5: 公式确认与拆解

**先列出论文中的核心公式，用 AskUserQuestion 让用户选择。**

使用 AskUserQuestion（多选）：

```
问题：论文中有以下核心公式/算法，你想拆解哪些？（可多选，选「其他」可输入你关注的其他公式或算法）
选项：
- "<公式1名称>" — <一句话说明>
- "<公式2名称>" — <一句话说明>
- "全部拆解" — 逐个讲解所有公式
- "跳过" — 不需要公式解读
multiSelect: true
```

#### 公式的双层输出

**核心问题**：Terminal 无法渲染 LaTeX（`$$` `$` 显示为原始符号），但笔记文件在编辑器中可以渲染。因此公式需要分两层写：

**对话中（Terminal 友好）**：用 ASCII / Unicode 写公式，确保终端里直接可读

```markdown
### <公式名称>

**公式**：Pass^k = (成功任务数 / 总任务数)，其中每个任务需 k 次独立运行全部成功

**直觉理解**：<一句大白话说这个公式在算什么>

**逐项拆解**：
- Pass^k：k 次运行全部通过的任务占比
- k：独立重复运行的次数，k 越大越严格
- 分子：k 次全部成功的任务数
- 分母：总任务数

**具体例子**：
假设有 100 个任务，每个任务跑 4 次：
- 有 60 个任务 4 次全部成功 → Pass^4 = 60/100 = 0.60
- 有 80 个任务至少 1 次成功 → Pass^1 = 80/100 = 0.80
```

**ASCII 公式写法规范**：
- 上标用 `^`：`Pass^k`、`x^2`
- 下标用 `_`：`S_world`、`A_agent`
- 分数用 `/` 或 `÷`：`成功数/总数`
- 求和用 `Σ`、乘积用 `Π`（Unicode）
- 张量积用 `⊗`
- 希腊字母用 Unicode 或英文：`τ` 或 `tau`
- 集合运算：`∈`、`∪`、`∩`
- 箭头：`→`、`←`、`↔`
- 不要使用 `$$...$$` 或 `$...$` 包裹

**笔记文件中（LaTeX 可渲染）**：保存时用标准 LaTeX 格式

```markdown
### <公式名称>

**公式**：
$$\text{Pass}^k = \frac{|\{t \in T : \text{all } k \text{ runs succeed}\}|}{|T|}$$

**直觉理解**：...

**逐项拆解**：
- $\text{Pass}^k$：k 次运行全部通过的任务占比
- $k$：独立重复运行的次数
- $T$：总任务集合
```

#### 公式步骤完成后的引导

输出完成后，追加到笔记文件，然后提示：

```
公式解读已保存到学习笔记。

💡 公式在终端里显示不太友好，打开笔记文件可以看到渲染效果更好的 LaTeX 版本：
   paper-notes/<name>/deep-learn.md

渲染方式：
- VS Code：打开文件后按 Cmd+Shift+V（Mac）/ Ctrl+Shift+V（Windows）预览
- Typora / Obsidian：直接打开即可渲染
- GitHub：push 后在网页端查看
```

**如果论文没有核心公式**，直接说明并跳过此步骤。

### Step 6: 附录精华提炼

**扫描论文附录，提取对理解和实践有价值的内容。**

```markdown
## 附录精华

论文附录包含以下内容，我为你提炼了关键部分：

### <附录章节标题>

**原始内容摘要**：<这部分附录在说什么>

**核心要点**（中文解读）：
<用中文把关键内容解释清楚，不是逐字翻译，而是提炼核心信息>

**实用价值**：<这部分内容对实际应用有什么用>
```

**提炼优先级**：
1. Prompt（提示词） / 配置模板 — 可直接复用到实际场景
2. 详细的实验设置 — 帮助复现
3. 补充的数据/分析 — 支持主文的结论
4. 纯理论推导 — 除非用户特别关注，否则简要提及

**输出完成后**：追加到笔记文件，提示已保存。

---

### Step 7: 实践指导

```markdown
## 实践指导

### 这篇论文的方法/发现可以怎么用？

**直接可用的**：
1. <可以直接拿来用的工具/数据/Prompt/配置>
2. ...

**需要适配的**：
1. <需要根据自己场景调整的方法/框架>
   - 适配建议：<怎么调整>
2. ...

**启发性的**：
1. <虽然不能直接用，但对思路/方向有启发的发现>
2. ...

### 行动建议

如果你想基于这篇论文做些什么，建议的优先级是：
1. **立刻可做**：<最小行动，今天就能开始>
2. **短期计划**：<一周内可以尝试的>
3. **长期方向**：<值得持续关注的>
```

**输出完成后**：追加到笔记文件，提示已保存。

## 全程可对话

学习模式的每个 Step 之间、以及 Step 内部，用户都可以随时用自然语言：

- **追问**："刚才说的 XXX 能再展开讲讲吗？"
- **补充术语**："帮我也解释一下 YYY"
- **修改笔记**："把笔记里 XXX 的解释改成 ..."
- **跳步**："公式那步跳过吧，直接到实践指导"
- **回溯**："回到术语那步，我还想加几个"

收到这类请求时：
1. 优先响应用户需求
2. 如果产生了新内容，更新到笔记文件
3. 然后继续之前的流程（或按用户指示跳转）

---

## 结束与提示

全部步骤完成后（或用户主动结束），输出：

```
学习笔记已完整保存到 paper-notes/<name>/deep-learn.md

你可以：
- 随时打开这个文件回顾学习内容
- 继续对话补充任何内容，我会实时更新到笔记中
- 切换到展示模式，准备一场基于这篇论文的汇报
```
